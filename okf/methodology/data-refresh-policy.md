# Data Refresh Policy

## Summary

The OKF bundle is built from a **point-in-time snapshot** of CricketStudio's data spine.
This file explains what that means for freshness and how often the bundle is refreshed.

## How Freshness Works

- The bundle's data comes from `cricketstudio-mcp/data/snapshot/*.json`, regenerated by
  CricketStudio's build pipeline. The current bundle uses the snapshot dated
  **2026-06-11** (see `manifest.yaml` → `dataset_version`).
- Each data-bearing claim carries a `provenance.computed_at` timestamp reflecting when the
  underlying value was computed — which may be earlier than the snapshot date.
- The **canonical CricketStudio page is always the freshest source.** During a live
  tournament, the website updates within hours of a match; the OKF bundle does not.

## Refresh Cadence

- **Between tournaments:** refresh the bundle when a new snapshot is published or when
  methodology changes.
- **During a live tournament:** the bundle is explicitly *not* a live source. Agents should
  defer to the canonical page for in-flight values and use OKF for definitions, scope, and
  context.

## Why This Matters

Treating a snapshot as live data is a correctness bug. The refresh policy makes the
staleness boundary explicit so agents disclose it.

## Agent Guidance

- State the `dataset_version` or `computed_at` when quoting a snapshot value.
- For "current" or "latest" questions during a live tournament, direct the reader to the
  canonical page.
